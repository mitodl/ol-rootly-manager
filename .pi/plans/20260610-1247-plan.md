## Context

The repository manages Rootly resources through Python data structures in `data.py` and an idempotent importer in `main.py`. The current Rootly functionality records for MIT Learn are incorrectly named: `MIT Learn AI` should display as `AskTim`, and `MITx Online` should display as `E-Commerce/Enrollment`. A new MIT Learn status page should be created using the Rootly API/importer without modifying the existing public or internal status pages.

## Goals / Non-goals

- Goals:
  - Create a new status page for MIT Learn.
  - Leave the existing status pages unchanged.
  - Display three functionalities on the new page:
    - `MIT Learn`
    - `AskTim`
    - `E-Commerce/Enrollment`
  - Connect functionality service membership by service-name prefixes:
    - `AskTim`: services named `MIT Learn AI - ...`
    - `E-Commerce/Enrollment`: services named `MITx Online - ...`
    - `MIT Learn`: services named `MIT Learn - ...` plus all `MITx Online - ...`, excluding `MIT Learn AI - ...`
  - Rename the incorrectly named existing functionality records in place rather than creating duplicate functionalities.
  - Avoid hard-coding functionality UUIDs into status page definitions where practical.

- Non-goals:
  - Do not update or replace the existing `Public System Status Page`.
  - Do not change unrelated Rootly resources.
  - Do not migrate this work to Pulumi as part of this change.

## Files to change

- `main.py` — enhance importer support so functionality renames can be matched by previous names/aliases and status pages can reference functionalities by name before sending Rootly-compatible payloads.
- `data.py` — update the MIT Learn-related functionality definitions, add the new `MIT Learn Status Page`, and attach the three functionalities to it.
- `README.md` — document any new data-file helper fields added for importer convenience, if the implementation introduces them.

## Ordered steps

1. Inspect the existing `FUNCTIONALITIES`, `STATUS_PAGES`, and service definitions in `data.py` to confirm current names and service-prefix memberships.
2. Update `main.py` functionality lookup/import logic to allow renaming an existing functionality in place, using an importer-only alias such as a previous-name list; strip that helper field before building Rootly SDK payloads.
3. Update `main.py` status page import logic to allow status pages to refer to functionality names, resolve those names to Rootly IDs during import, and strip importer-only helper fields before sending payloads.
4. Update `data.py` so:
   - `MIT Learn AI` becomes `AskTim` while preserving in-place rename metadata.
   - `MITx Online` becomes `E-Commerce/Enrollment` while preserving in-place rename metadata.
   - `MIT Learn` service membership uses the agreed prefix rules.
   - A new status page is added with title `MIT Learn Status Page`, public title `MIT Learn | Status`, website URL `https://learn.mit.edu`, enabled/public settings, and the three functionality names attached.
5. Ensure the existing public and internal status page entries remain present and unchanged except for formatting if unavoidable.
6. Update `README.md` only if new importer-only fields need documentation for future maintainers.

## Validation

- Run Python syntax/type sanity checks appropriate for this repo, such as `uv run python -m py_compile main.py data.py`.
- Run the importer/report flow in a safe way before applying changes, if available, to confirm the new status page and functionality names resolve as expected.
- Run `uv run python main.py --import data.py` when ready to apply to Rootly.
- After import, verify in Rootly or via `uv run python main.py --report` that:
  - A new `MIT Learn Status Page` exists.
  - Existing status pages are still present and unchanged.
  - Functionalities display as `MIT Learn`, `AskTim`, and `E-Commerce/Enrollment`.
  - The functionality service memberships match the agreed prefixes.

## Risks & unknowns

- Rootly API behavior around renaming functionalities may differ from the SDK model assumptions; the importer should report any non-200 response clearly.
- The exact Rootly status page fields required for a new page may need to be copied/minimized from existing examples to satisfy API validation.
- Service prefix matching depends on current service names remaining consistent.
- If Rootly requires status pages to use direct functionality IDs only, name resolution must happen at import time before payload submission.
